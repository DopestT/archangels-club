# Archangels Club — Bug Squash System

Automated checks, structured bug reports, and safe auto-fixes.

---

## Quick reference

| Command | What it does |
|---|---|
| `npm run check` | TypeScript + ESLint (fast, no network) |
| `npm run typecheck` | TypeScript check on both server and client |
| `npm run lint` | ESLint on both server and client |
| `npm run test` | Backend unit tests (Vitest, no real DB needed) |
| `npm run test:api` | Live API health checks against Railway |
| `npm run test:e2e` | Playwright E2E tests against production |
| `npm run bug:squash` | Full pipeline: typecheck → lint → test → build → API → E2E |
| `npm run bug:report` | Run all checks, output `bug-report.json` + summary |
| `npm run bug:fix` | Read last report, apply safe automatic fixes |

## Running locally

### Prerequisites

```bash
# From the project root
cd server && npm install
cd ../client && npm install
# Install Playwright browser (one-time)
cd client && npx playwright install chromium
```

### Running individual steps

```bash
# Type check only
npm run typecheck

# Lint only
npm run lint

# Backend unit tests (no DB or network required)
npm run test

# API health checks (hits Railway — needs internet)
npm run test:api

# E2E tests (hits archangelsclub.com — needs internet + Playwright installed)
npm run test:e2e

# Full pipeline (fail-fast — stops at first failure)
npm run bug:squash
```

### Override URLs

```bash
# Point API health checks at a local server
API_BASE_URL=http://localhost:4000 npm run test:api

# Point Playwright at a local frontend
PLAYWRIGHT_BASE_URL=http://localhost:3000 npm run test:e2e

# Both at once
API_BASE_URL=http://localhost:4000 PLAYWRIGHT_BASE_URL=http://localhost:3000 npm run bug:squash
```

### Run schema tests against Railway DB

```bash
DATABASE_URL=<railway-public-url> npm --prefix server run test
```

## Bug Report (`npm run bug:report`)

Unlike `bug:squash`, the report runner **does not stop on first failure** — it runs all five checks and collects every error into a single JSON file.

### Output

**`bug-report.json`** (written to project root):

```jsonc
{
  "timestamp": "2026-04-26T05:20:00Z",
  "git_sha": "4a5aab4",
  "summary": { "total": 2, "errors": 1, "warnings": 1, "all_pass": false },
  "checks": [
    { "name": "typecheck",     "pass": true  },
    { "name": "lint",          "pass": false },
    { "name": "backend-tests", "pass": true  },
    { "name": "frontend-build","pass": true  },
    { "name": "api-health",    "pass": false }
  ],
  "errors": [
    {
      "id": "err-1",
      "type": "lint",
      "severity": "warning",
      "file": "server/src/routes/admin.ts",
      "line": 42,
      "code": "@typescript-eslint/no-unused-vars",
      "message": "'result' is assigned but never used",
      "fix_category": null,
      "suggestion": "ESLint rule @typescript-eslint/no-unused-vars. Run: eslint --fix"
    },
    {
      "id": "err-2",
      "type": "api-health",
      "severity": "error",
      "route": "GET /api/content?sort=trending",

```

### Error types

| `type` | Source | `fix_category` values |
|---|---|---|
| `typescript` | `tsc --noEmit` output | `null-check` or null |
| `lint` | ESLint JSON output | null |
| `test-failure` | Vitest output | `missing-column`, `undefined-route`, or null |
| `sql-error` | PostgreSQL error in test output | `missing-column` |
| `build` | Vite build output | null |
| `api-health` | `check-api.mjs` failures | `missing-column`, `undefined-route`, or null |

## Bug Fix (`npm run bug:fix`)

Reads `bug-report.json` and applies three categories of safe, mechanical fixes.

### What it fixes automatically

#### `missing-column` → ALTER TABLE migration

Writes a `.sql` file to `server/src/db/patches/`:

```sql
-- fix-2026-04-26T05-20-00-add-content_unlocks-created_at.sql
ALTER TABLE content_unlocks ADD COLUMN IF NOT EXISTS created_at TIMESTAMPTZ DEFAULT NOW();
UPDATE content_unlocks SET created_at = NOW() WHERE created_at IS NULL;
```

**To apply:**
```bash
psql "$DATABASE_URL" -f server/src/db/patches/<filename>.sql
```

The patch directory is `.gitignore`d by default. Review the file before committing it as a permanent migration step in `server/src/db/migrate.ts`.

#### `undefined-route` → stub handler

Appends a 501 stub to the correct route file:

```typescript
// STUB: auto-generated by npm run bug:fix — replace with real implementation
router.get('/some-path', (_req, res) => {
  res.status(501).json({ error: 'Not implemented' });
});
```

#### `null-check` → suggestion only

Null-check fixes are context-dependent (the right fix is `if (!x) return`, `x?.prop`, or a different type guard depending on the code). `bug:fix` prints the file and line but does not edit the file.

### What it never touches

- Business logic (payment flows, pricing, unlock logic)
- Auth middleware
- Database migrations that already exist
- Any file not directly identified by the error

## What each check tests

### TypeScript (`npm run typecheck`)

Runs `tsc --noEmit` on both `server/` and `client/`. Catches:
- Missing properties on types
- Null/undefined access (`Object is possibly null`)
- Wrong argument types
- Unreachable code

### ESLint (`npm run lint`)

Runs ESLint with `typescript-eslint` recommended rules. Catches:
- Unused variables
- React hook violations (client)
- `@typescript-eslint` rule violations

### Backend unit tests (`npm run test`)

Vitest tests in `server/src/tests/`. No real database or network needed — DB is mocked with `vi.mock`.

| Test file | What it checks |
|---|---|
| `routes.test.ts` | `GET /api/creators` returns array; `GET /api/content` returns array; locked content strips `media_url`; payment endpoint rejects unauthenticated; admin approval handles duplicate usernames without 500 |
| `schema.test.ts` | Verifies required columns exist in the real DB (skipped automatically if `DATABASE_URL` is not set) |

### Frontend build (`npm --prefix client run build`)

Runs `tsc && vite build`. Catches import errors, missing types, and broken module references that TypeScript alone wouldn't catch.

### API health checks (`npm run test:api`)

Hits the live Railway deployment. Checks:
- `/api/health` → `{ status: "ok" }`
- `/api/health/db` → database connected
- `/api/creators` → non-empty array (verifies demo seed ran)
- `/api/content`, `?sort=trending`, `?sort=rising` → arrays (catches SQL column errors)
- `/api/activity/recent` → array
- `demo-c-aria-01` content → `media_url: null` (verifies locked content is not leaking media)
- `/api/payments/create-unlock-session` → 401 unauthenticated
- `/api/auth/login` with bad credentials → 400 or 401

### Playwright E2E tests (`npm run test:e2e`)

Runs against the production frontend. Tests:
- Homepage loads, shows CTA, no console errors
- Explore loads creator cards, API returns creators
- Creator profile shows name, subscribe button, tip button
- Locked content shows paywall; unlock button redirects to login
- `media_url` not visible in API response for locked content
- Subscription upsell does not claim "full access" or "unlimited"
- Admin page redirects to login unauthenticated
- Auth guards active on payment endpoint

## GitHub Actions

Every push to every branch runs the full pipeline. The workflow is at `.github/workflows/bug-squash.yml`.

On failure:
- `bug:report` runs automatically and the `bug-report.json` is uploaded as an artifact named `bug-report-<sha>`
- Playwright traces and screenshots are uploaded as `playwright-report-<sha>`
- Artifacts are kept for 14 days

To download a report from a failed run:
1. Go to the Actions tab in GitHub
2. Click the failed run
3. Scroll to Artifacts at the bottom
4. Download `bug-report-<sha>.zip`

## When a check fails

| Failure | First thing to check |
|---|---|
| TypeScript fails | Read the error line. Most common: null access (`TS2531`) or wrong type argument (`TS2345`). |
| ESLint fails | Run `npm run lint` locally and read the rule ID. Most rules auto-fix with `eslint --fix`. |
| Backend tests fail | Run `npm run test` locally. If DB-related: `DATABASE_URL=<railway-url> npm run test`. |
| Build fails | Read the Vite error. Usually a broken import or a TypeScript error not caught by `typecheck`. |
| API health fails with "column … does not exist" | Run `npm run bug:report && npm run bug:fix` — a migration patch will be generated. |
| API health fails with "DB not connected" | Check Railway dashboard — the service may be sleeping or crashed. |
| E2E tests fail | Check if the production site is up at archangelsclub.com. Then check if the specific page/element changed. |
| E2E "media_url leaked" | A `GET /api/content/:id` handler is returning `media_url` for non-free content. Check `content.ts`. |

## Adding new tests

### Backend (Vitest)

Add test cases to `server/src/tests/routes.test.ts`. If you add a new route file, mock any new service imports at the top of the test file with `vi.mock(...)`.

### E2E (Playwright)

Add spec files to `client/tests/e2e/`. Tests run against the production URL — keep them read-only (no data modification). Use `apiRequest` for direct API assertions without the browser.

### API health

Add new `check()` calls to `scripts/check-api.mjs`. Follow the existing pattern: describe what you expect, throw with a descriptive message on failure.
